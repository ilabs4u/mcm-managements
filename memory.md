## 1. Project Overview

MCM is a biryani franchise management system. The **main franchise owner** supplies raw materials to **30+ franchise outlets**. The owner needs to track daily material consumption across all franchises.

**Core Problem**: Franchise managers currently manually calculate material usage — error-prone and tedious.

**Core Solution**: Managers only log *what they produced* (kg). The system auto-calculates all raw material usage (rice, berista, masala) based on recipes the owner defines.

**Users**:
- **Owner** (1): Sees all franchises, manages recipes, approves new franchises, views reports
- **Franchise Managers** (~30): Log daily production, view their history

**Key Entities**: Products (Dum Biryani, Tandoori Biryani, Tikka), Ingredients (Rice, Berista, Masalas), Franchises, Daily Logs, Production Entries

## 7. Backend Architecture

### API Pattern
All responses: `{ success: boolean, data?: T, error?: string }`

### API Groups
- **Auth** (4 endpoints): register, login, logout, me
- **Owner** (5 endpoints): dashboard, franchises list, franchise detail, approve/reject, reports
- **Franchise** (5 endpoints): today's log, add entry, delete entry, toggle open, history
- **Products** (4 endpoints): list, create, update, update recipe

### Server-Side Calculation
When a franchise entry is created, the server:
1. Fetches the latest recipe from DB (source of truth)
2. Calculates packets = quantity_kg / packet_size_kg
3. Calculates material breakdown from recipe × packets
4. Stores the breakdown as a JSON snapshot in `daily_log_items.material_breakdown`

This ensures recipe changes only affect future entries, not past ones.

## 8. Database Architecture

**7 tables** in Supabase PostgreSQL with Row-Level Security:

| Table | Purpose | Key Constraint |
|-------|---------|---------------|
| profiles | User data + role | 1:1 with auth.users |
| franchises | Franchise outlets | UNIQUE manager_id |
| products | Product catalog | 3 seed products |
| ingredients | Raw materials | 4 seed ingredients |
| product_ingredients | Recipe mappings | UNIQUE(product, ingredient) |
| daily_logs | Daily container | UNIQUE(franchise, date) |
| daily_log_items | Production entries | Multiple per log |

**RLS Rules**:
- Franchise managers see only their own data
- Owner sees all data
- Products/ingredients readable by all, writable by owner only

## 9. Authentication Flow

```
Register → (Supabase Auth creates user) → profiles(franchise_manager) → franchises(pending)
    ↓
Login → Middleware checks: role + franchise status
    ↓
If owner → /owner
If manager + approved → /franchise
If manager + pending → /pending
If manager + rejected/suspended → error message
```

## 10. Data Flow Diagrams

### Core Flow: Production Entry
```
Manager types 1.5 kg Dum Biryani
    ↓
Client: calculations.js shows live preview
    (0.5 pkt, 2.0kg rice, 250gm berista, 0.5pkt masala)
    ↓
Manager clicks "Add Entry"
    ↓
Server: fetches latest recipe, recalculates
    ↓
DB: UPSERT daily_logs + INSERT daily_log_items
    ↓
Supabase Realtime broadcasts event
    ↓
Owner dashboard updates live
```

### Auto-Calculation Formula
```
packets = quantity_kg / packet_size_kg
ingredient_used = packets × ingredient.quantity_per_packet
```

## 16. Known Risks

- **Supabase vendor lock-in** — auth + realtime + RLS tied to Supabase
- **Free tier limits** — 500MB DB, auto-pausing, no backups
- **Single owner account** — no multi-owner support in Phase 1
- **No rate limiting** — API routes exposed without throttling
- **Material breakdown snapshots** — if recipe changes, historical data stays frozen (by design, but could confuse users)
